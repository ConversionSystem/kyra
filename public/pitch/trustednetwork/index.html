Line-by-Line Assessment

Each SOW deliverable graded on complexity, accuracy, and whether it's being over- or under-scoped.

A. OpenClaw Deployment Easy

"Install or rebuild OpenClaw. Fix all configuration issues. Confirm API and gateway connections work."

Reality: OpenClaw installs in one command (npm install -g openclaw or Docker). Configuration is a single JSON file. Gateway auto-starts. This is 30 minutes of work being billed as an entire milestone.

On Kyra: Auto-provisioned container at account creation. Zero setup required.

B. Microsoft 365 Integration Hard

"Connect OpenClaw to Microsoft 365 using Graph API: Outlook Email, OneDrive, Teams."

Reality: This is the hardest item in the entire SOW. OpenClaw has no native M365 plugin. Building full Graph API integration requires: Azure AD app registration, OAuth2 flows, token refresh logic, permission scoping, and custom code for each service.

Estimated effort: 40-80 hours of custom development.

The SOW gives this ONE LINE — "M365 integration complete" — for what is realistically weeks of work. If the contractor prices this cheaply, corners will be cut.

On Kyra: 3 dedicated AI workers (Outlook, OneDrive, Teams) sharing one Azure AD app. Each worker has specific behaviors — email triage, file management, channel monitoring. Token refresh handled automatically by the platform.

C. Fathom Integration Medium

"Connect to Fathom to pull meeting transcripts, summaries, and action items."

Reality: Fathom's API is limited/in beta. Most reliable method is webhook or email-based — Fathom sends summary after each meeting, integration parses it. "Structured data (JSON)" output requires custom AI parsing since Fathom outputs prose, not JSON.

Estimated effort: 15-25 hours.

On Kyra: AI worker receives meeting data via webhook, email forward, or API. Extracts attendees, decisions, action items with owners and deadlines. Outputs structured JSON. Cross-posts summaries to Teams. Follows up on overdue action items.

D. Telegram Bot Trivial

"Set up a secure Telegram bot. Handle inbound and outbound messages. Token rotation, encrypted storage, rate-limit protection."

Reality: Telegram is a native OpenClaw channel plugin. Setup: create bot via @BotFather → paste token into config. Done. Encrypted storage, rate limiting, and message handling are all built into the platform.

Actual effort: 5 minutes.

The SOW describes this as if it requires custom security work. It doesn't — it's a config entry.

On Kyra: Native channel. Paste bot token in dashboard. Instant.

E. Web Search Trivial

"Integrate Brave Search API (or equivalent) for live search. Content retrieval, safe scraping controls, request throttling."

Reality: Web search is a built-in OpenClaw tool. Add Brave or Perplexity API key → search works. Content retrieval, throttling, and safe scraping are all platform features.

Actual effort: 2 minutes.

On Kyra: Built-in web_search tool. Add API key in Secrets Vault. Done.

F. GitHub Integration Easy

"Support branch creation, commits, pull requests, GitHub Actions triggers, and automated deployments."

Reality: OpenClaw has gh CLI pre-installed. Authenticate with a personal access token and every GitHub operation works immediately — branches, commits, PRs, Actions triggers, deployment monitoring.

Actual effort: 5 minutes to authenticate.

On Kyra: Add GitHub token to Secrets Vault. AI worker handles all git operations via natural language commands.

What the SOW Misses

Critical gaps that will cause problems after delivery — things the SOW doesn't mention but the system needs.

  • ✕No ongoing maintenance. OAuth tokens expire. APIs change. Without maintenance, integrations will break silently within months. The SOW explicitly excludes this.
  • ✕No token refresh logic. M365 OAuth tokens expire every 60-90 minutes. The SOW doesn't mention refresh handling. Without it, email/OneDrive/Teams will stop working regularly.
  • ✕No health monitoring. If the system goes down at 2am, who knows? No uptime monitoring, no auto-restart, no alerting mentioned anywhere.
  • ✕No AI behavior configuration. The SOW connects pipes — it doesn't make the system intelligent. There's no mention of how the AI worker should triage emails, what to do with meeting transcripts, or how to handle exceptions.
  • ✕No backup or disaster recovery. If the VPS dies, everything is lost. No backup strategy mentioned.
  • ✕IP goes to MIX Networks, not client. Section 7 states: "All work product belongs to MIX Networks upon payment." The client pays but doesn't own the code. If they want changes later, they need MIX Networks' permission.
  • ✕No scalability plan. What happens when you add more integrations? Each one is a new SOW, new negotiation, new development cycle.

Security Assessment

The SOW's security section is its strongest part. Here's how each requirement maps to Kyra.

  • ✓OAuth-based authentication — Azure AD for M365, token-based for others. Kyra handles OAuth flows and token refresh automatically.
  • ✓Credentials encrypted at rest — Kyra's Secrets Vault stores all API keys and tokens encrypted. Not in environment variables or plain config files.
  • ✓Minimum necessary permissions — Each skill gets only the Graph API scopes it needs. M365 skill uses delegated permissions, scoped to the specific operations required.
  • ✓Rate limiting on all API calls — Built into the Kyra gateway. Applies to all external requests automatically.
  • ✓Action logging — Every operation logged with timestamps, actor, and context. Visible in the dashboard's conversation feed and analytics.
  • ✓Isolated containers — Your instance runs in its own Docker container, not shared with other clients. Network-isolated, resource-limited.
  • ✓Auto token refresh — Not in the SOW but critical. Kyra handles M365 token refresh before expiry so integrations never silently break.

Risk Analysis

Risks of proceeding with the SOW as written vs. using Kyra.

SOW Risks
  • ✕Blank pricing + timeline. No way to evaluate cost or schedule. The contractor could stretch delivery indefinitely.
  • ✕Single point of failure. One contractor builds everything. If they leave, no one can maintain it.
  • ✕No maintenance = silent failures. OAuth tokens expire. APIs change. Without ongoing support, you'll discover things are broken when you need them most.
  • ✕IP lock-in. MIX Networks owns the code. Switching contractors or making changes requires their permission.
  • ✕Under-scoped M365. The hardest integration (40-80 hrs) gets one line in the SOW. This will either be delivered half-baked or blow the timeline.

Recommendation

The SOW covers the right integrations but approaches them as a custom development project. 5 of 8 deliverables (OpenClaw deployment, Telegram, web search, GitHub, security) are already built into the platform — they're configuration, not code.

The real work is Microsoft 365 and Fathom. On Kyra, these become skills on a single AI worker — not just API connections, but intelligent capabilities that let one worker triage emails, parse meetings, manage files, and post to Teams through natural language. One brain, shared context across every tool.

The critical gap in the SOW is maintenance. Without ongoing token refresh, health monitoring, and platform updates, the system will degrade. Kyra includes all of this as part of the platform — not as a separate contract negotiation.
